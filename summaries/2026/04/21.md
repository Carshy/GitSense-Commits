The project "Collectible Trading Post" demonstrates a hybrid rendering architecture, utilizing an Express.js backend for API services and an Express.js frontend for serving EJS page shells, which are then hydrated with interactive Lit components. **Key Architectural Decisions (from `ADR.md`):**
*   **Hybrid Rendering:** EJS is used for server-side rendering of the page shell, while Lit handles client-side interactive components, with the server determining component placement (`component plan`).
*   **Component Plan:** A server-side layer (`component-plan.js`) defines which Lit components and their initial properties should be rendered for each route (e.g., home page gets hero, search, item grid; item detail page gets detail, offer panel, chat, checkout).
*   **HTTP Polling for Chat:** Instead of websockets, HTTP polling is used for live message updates, aligning with the starter backend's capabilities and simplifying implementation.
*   **File-based JSON Persistence:** The backend uses JSON files for data storage (items, users, messages) rather than a database, for rapid iteration and to align with the provided starter code.
*   **Frontend API Proxying:** The frontend server proxies `/api/*` requests to the backend server, maintaining a single origin for client-side API calls and simplifying component code.

**Patterns and Recurring Elements:**
*   **Consistent Eventing:** Components communicate using `CustomEvent`s like `marketplace-search-change`, `marketplace-item-refresh`, `marketplace-thread-refresh`, `marketplace-seller-dashboard-refresh`, and `marketplace-toast` for propagating state changes and user feedback across the application.
*   **Modular API Client:** The `api-client.js` centralizes fetch requests, promoting reusability and consistent error handling for backend interactions.
*   **Styling Consistency:** There's a clear pattern of migrating component styles from inline Tailwind classes or simple CSS to using a set of defined CSS variables (`--brand-navy`, `--brand-line`, etc.) for a cohesive design system, particularly evident from 4/20/2026, 11:11:06 AM onwards.
*   **Conditional Loading and Error States:** Lit components consistently implement `loading` and `error` states, rendering appropriate UI messages to the user during data fetching or in case of failures.
*   **Lifecycle Management:** Components increasingly use `connectedCallback`, `disconnectedCallback`, and `updated` lifecycle hooks to manage event listeners and data loading efficiently, especially when component properties (`itemId`, `sellerId`) change.
*   **User Context (`viewerId`, `viewerName`):** These properties are frequently passed to components related to item details, messages, and offers, indicating a design choice to allow testing or representing different user roles within the same UI.
