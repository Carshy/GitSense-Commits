### Patterns and Recurring Elements:

1.  **Monorepo Structure**: The project (`social-marketplace`) is set up as a monorepo with `server` and `web` sub-projects, managed by `concurrently` for parallel development.
2.  **Hybrid Rendering**: A core theme is the implementation of a hybrid rendering approach. EJS templates (`.ejs` files) are used for server-side rendering the initial HTML shell, which then hosts client-side Lit components.
3.  **Lit Components**: Extensive use of Lit for interactive UI elements. Components often follow a pattern of:
    *   Defining `static properties` for data binding.
    *   Using `static styles` for scoped CSS.
    *   Implementing `connectedCallback` and `disconnectedCallback` for event listener management.
    *   `load...` methods (e.g., `loadItems`, `loadThread`) for data fetching, often with loading/error states.
    *   `render()` method for declarative UI.
    *   Custom events (`marketplace-search-change`, `marketplace-item-refresh`, `marketplace-thread-refresh`, `marketplace-seller-dashboard-refresh`) are used for inter-component communication and data synchronization.
4.  **Data Persistence (Backend)**: JSON files (`items.json`, `users.json`, `messages.json`) are used for simple data persistence, managed by `server/lib/db.js` utilities.
5.  **API Endpoints**: The backend defines clear API endpoints (`/api/items`, `/api/users`, `/api/messages`) for managing marketplace data.
6.  **Validation**: A dedicated `server/lib/validators.js` file centralizes input validation for API payloads.
7.  **Authentication/Authorization**: Basic session-based authentication is implemented in the backend (`express-session`, `server/lib/auth.js`, `server/routes/users.js`) with password hashing.
8.  **Real-time Interaction (Polling)**: The `marketplace-chat-thread` component implements client-side polling for new messages, mimicking real-time updates.
9.  **Styling**: Initial styling uses basic Tailwind classes. A significant later change introduces a comprehensive set of CSS variables in `web/src/styles/input.css` and migrates EJS and Lit component styling to use these variables, enhancing design consistency.
10. **Development Workflow**: The `package.json` scripts reflect a dev-focused workflow (`npm run dev`) that starts both server and web projects, including CSS compilation.
11. **Refactoring and Modularity**: Early monolithic structures (e.g., in `web/src/server/app.js`) are refactored into smaller, more focused modules (e.g., `component-plan.js`, `view-models`, `routes/pages.js`).
12. **Incremental Component Implementation**: The `main.js` and `component-plan.js` initially use generic "placeholder" Lit components, which are progressively replaced with actual functional Lit components (`marketplace-item-detail`, `marketplace-offer-panel`, etc.) throughout the log.
